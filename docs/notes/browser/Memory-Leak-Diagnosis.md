# 内存泄漏排查入门：哪些引用没有被释放

单页应用打开得越久越卡，未必是「浏览器内存不够」，也可能是已经离开的页面仍被某个引用保留。内存泄漏的关键不在于对象存在，而在于它本该失去引用却仍然可达。

![内存泄漏排查流程：重复同一操作，拍摄并比较堆快照，沿保留路径找到未清理的引用。](/illustrations/frontend/memory-leak-diagnosis.svg)

## 先构造可重复的现象

不要只看任务管理器里某一个瞬时数字。打开 Chrome DevTools 的 **Memory** 面板，按相同路径重复进入和离开目标页面，例如「进入详情页 -> 返回列表」执行 10 次。分别在操作前后拍 Heap Snapshot，并强制垃圾回收后比较：如果本应消失的组件、DOM 节点或数据数组持续增长，就值得继续排查。

## 最常见的三类引用

### 1. 未移除的事件监听器

全局对象上的监听器会长期存在，回调闭包可能持有组件状态。组件销毁时必须成对清理。

```js
function mount() {
  const onResize = () => updateLayout();
  window.addEventListener('resize', onResize);

  return () => {
    window.removeEventListener('resize', onResize);
  };
}
```

不要在 `removeEventListener` 时重新创建一个函数；只有同一个函数引用才能被移除。

### 2. 未清除的定时器和订阅

轮询、`setInterval`、WebSocket 订阅和观察者在页面离开后仍可能继续运行。把创建和清理放在同一个生命周期里，避免清理逻辑散落在多个分支。

```js
const timer = setInterval(refresh, 5_000);

function unmount() {
  clearInterval(timer);
}
```

### 3. 无上限缓存或闭包引用

缓存没有淘汰策略时会无限增长；闭包若持有大数组或已删除 DOM 节点，也会阻止垃圾回收。缓存应限定大小、过期时间或按路由清理，避免无意义地把对象挂在模块全局或 `window` 上。

## 用保留路径找到根因

在堆快照中选中持续存在的对象，查看 **Retainers**（保留路径）。它展示对象为何仍可达：可能是监听器回调、定时器、全局 Map，也可能是框架开发模式下的调试引用。修复的目标是移除不该存在的上游引用，而不是在下游手动置空一个仍被引用的对象。

修复后重复同一组操作并再次比较快照。对象数量应回到稳定水平，而不是只凭一次刷新后的主观感受判断。

## 常见误区

**内存曲线向上就是泄漏。** 垃圾回收是分批进行的。应在相同操作和垃圾回收后比较多个快照。

**把变量设为 `null` 就能释放。** 只要还有其他引用，垃圾回收器不会回收它。

**框架会自动清理一切。** 框架通常能清理组件自身资源，但全局监听器、第三方库订阅和模块级缓存仍由应用负责。

## 小结

排查内存问题的可靠路径是：复现 -> 快照 -> 比较 -> 查看保留路径 -> 移除上游引用 -> 再验证。优先检查监听器、定时器、订阅和无界缓存，能覆盖大多数前端泄漏问题。
